# تقرير تقييم مشروع Shahenco

تاريخ التقييم: 2026-06-30

## الخلاصة التنفيذية

المشروع يحتوي على عمل فعلي لتطبيق إدارة شحن بضائع: باك إند Django، لوحة إدارة ويب، واثنين على الأقل من تطبيقات Flutter. التقييم التقني الحالي: 5/10.

المشروع مناسب كنسخة MVP أو مشروع داخلي قابل للتطوير، لكنه غير جاهز للإنتاج كما هو بسبب مخاطر أمان، تكرار في الكود، ضعف في إدارة الاعتمادات، وغياب اختبارات وتشغيل موثق.

## نقاط القوة

- الدومين غني ويغطي كيانات مهمة: مستخدم، مندوب، كابتن، شركة توصيل، سيارة، طلب، عرض، تقرير، دفع، غرامات، حضور، إشعارات، شات.
- وجود Django REST Framework وواجهات API.
- وجود Channels/WebSocket مبدئيا للشات أو التحديثات اللحظية.
- وجود لوحة إدارة ويب واسعة مع CRUD وExcel exports.
- تطبيقات Flutter تستخدم خرائط، Firebase Messaging، إشعارات، رفع صور، ماسح باركود/QR، وتخزين محلي.

## المخاطر العالية

### 1. إعدادات أمان غير مناسبة للإنتاج

في إعدادات Django:

- `ALLOWED_HOSTS = ['*']`
- CORS مفتوح عبر `CORS_ORIGIN_ALLOW_ALL = True`
- `CORS_ALLOW_CREDENTIALS = True`
- في نسخة `shahen`: `DEBUG = True`
- `SECRET_KEY` له fallback عام: `dev-secret-key`

الأثر: التطبيق قد يكون معرضا لتسريب أخطاء، قبول طلبات من نطاقات غير موثوقة، ومشاكل أمنية عند النشر.

### 2. API عام بدون حماية واضحة

المسارات العامة تستخدم نمطا مثل:

`api/v1/<str:app>/<str:model>`

وفي `GenericAutoView` توجد:

- `authentication_classes = []`
- `permission_classes = []`
- توليد serializer عام مع `fields = '__all__'`

الأثر: إذا لم توجد حماية خارجية قوية، قد يتم كشف أو تعديل نماذج حساسة مباشرة.

### 3. بيانات دخول داخل السورس

تم العثور على كلمات مرور في:

`D:/shahenco_folder/Programmers/Source code/New Text Document.txt`

الأثر: أي تسليم أو مشاركة للمشروع قد تسرب بيانات دخول.

### 4. قاعدة بيانات SQLite داخل المشروع

يوجد `db.sqlite3` داخل نسخ الباك إند.

الأثر: غير مناسب للإنتاج، وقد يحتوي بيانات حقيقية، ويصعب إدارة النسخ والنسخ الاحتياطي.

## مخاطر متوسطة

### 1. تكرار نسخ الباك إند

يوجد باك إند شبه مكرر في `uae` و`shahen`. هذا يخلق مشكلة: أي إصلاح أمني أو ميزة يجب تطبيقها مرتين، وقد تختلف النسختان بمرور الوقت.

### 2. ملفات لا يفترض وجودها في التسليم

تم العثور على:

- `.venv`
- `.idea`
- `.git`
- `build`
- `staticfiles`
- `media`
- `__MACOSX`
- `.DS_Store`
- ملفات `.zip` داخل مجلد السورس

الأثر: حجم كبير، فوضى في التسليم، احتمال تسريب ملفات محلية أو بيانات مستخدمين.

### 3. الاعتمادات غير مثبتة بإصدارات

في `requirements.txt` أغلب الحزم بلا إصدارات. وفي `pubspec.yaml` توجد حزم Flutter كثيرة بلا version constraints.

الأثر: قد يعمل المشروع اليوم ويفشل لاحقا عند تثبيت إصدار أحدث من نفس الحزم.

### 4. تجاهل migrations في إحدى النسخ

في `.gitignore` الخاص بنسخة `shahen` توجد قواعد تتجاهل migrations لبعض التطبيقات.

الأثر: تشغيل المشروع على جهاز جديد أو سيرفر جديد قد يفشل أو ينتج قاعدة بيانات غير متطابقة.

## خطة الإصلاح المقترحة

### المرحلة 1: تنظيف وتسليم منظم

1. اختيار النسخة المعتمدة من الباك إند: `uae` أو `shahen`.
2. إزالة ملفات البناء والبيئات والنسخ المضغوطة من التسليم.
3. إنشاء بنية واضحة:
   - `backend/`
   - `apps/captain/`
   - `apps/mandob/`
   - `docs/`
4. تحديث `.gitignore` وتأكيد عدم تتبع الملفات الثقيلة أو الحساسة.

### المرحلة 2: أمان وتشغيل

1. نقل كل الأسرار إلى `.env`.
2. ضبط `DEBUG` من متغير بيئة.
3. تحديد `ALLOWED_HOSTS`.
4. إغلاق CORS على نطاقات محددة فقط.
5. إضافة إعدادات production مثل HTTPS cookies وCSRF.
6. استبدال SQLite ب PostgreSQL للإنتاج.

### المرحلة 3: API وصلاحيات

1. مراجعة `GenericAutoView`.
2. منع الوصول العام للنماذج الحساسة.
3. تحديد serializers صريحة لكل نموذج مهم.
4. إضافة authentication/permissions افتراضية في DRF.
5. مراجعة token handling وتخزينه.

### المرحلة 4: جودة وتوثيق

1. تثبيت إصدارات `requirements.txt` و`pubspec.yaml`.
2. كتابة README تشغيلي.
3. إضافة اختبارات للآتي:
   - login لكل نوع مستخدم
   - إنشاء طلب
   - قبول/رفض عرض
   - صلاحيات لوحة الإدارة
   - رفع الملفات
4. إضافة أوامر تشغيل موحدة.
